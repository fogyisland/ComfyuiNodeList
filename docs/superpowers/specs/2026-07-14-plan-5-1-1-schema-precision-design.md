# Plan 5.1.1 — Fix `gitsha_resolutions.resolved_at` precision

**Date:** 2026-07-14
**Status:** Approved (brainstormed with user 2026-07-14, scope confirmed: Option A — minimum)
**Parent plan:** Plan 5.1 (target_commitish resolution), §Followups "Plan 5.1.1 candidate"

## Problem

`web/prisma/schema.prisma:165` declares:

```prisma
  resolved_at DateTime @default(now()) @db.DateTime
```

Prisma maps `@db.DateTime` to bare `DATETIME` (precision 0, second-level). The generated migration in `web/prisma/migrations/20260712_gitsha_resolutions/migration.sql` is:

```sql
  `resolved_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
```

**Why this matters:**

1. **Inconsistency with baseline.** Every other `@default(now())` column in the project (e.g., `users.created_at`, `nodes.created_at`, `scan_failures.occurred_at`) resolves via Prisma's default `DateTime` mapping to `DATETIME(3)` (millisecond precision) — see `web/prisma/migrations/20260621122750_init/migration.sql` lines 9, 24, 39, 76, 90. The `resolved_at` column is the only `@default(now())` column at precision 0.

2. **Historian / observability loss.** Second-level precision is too coarse for cache-TTL logic that compares `NOW() - resolved_at` against 7-day boundaries and for any future cache-hit metrics. Production may not yet need ms precision but adding it now (when the table is empty in production) is free; adding it later (when the table has rows) requires a migration too.

3. **Source of historical noise.** Per Plan 5.1 smoke-test doc and the Plan 5.1 spec, this column was the trigger for the 4-commit cascade that migrated test fixtures from `prisma db push --force-reset` to `prisma migrate deploy`. The fix is metadata-only; aligning precision with the rest of the project makes any future test-fixture decision safer.

## Solution

Change the schema annotation from `@db.DateTime` to `@db.DateTime(3)` and add one ALTER migration that promotes the existing column's precision to match.

## Design

### Component 1: Schema change

In `web/prisma/schema.prisma`, line 165:

```prisma
model GitShaResolution {
  id          Int      @id @default(autoincrement())
  owner       String   @db.VarChar(255)
  repo        String   @db.VarChar(255)
  ref         String   @db.VarChar(255)
  sha         String   @db.Char(40)
- resolved_at DateTime @default(now()) @db.DateTime
+ resolved_at DateTime @default(now()) @db.DateTime(3)

  @@unique([owner, repo, ref])
  @@index([resolved_at])
  @@map("gitsha_resolutions")
}
```

Rationale: explicit `@db.DateTime(3)` matches Prisma's default mapping for `DateTime` without `@db.X`, so it's indistinguishable from omitting it — but spelling it out documents intent and survives any future Prisma default-mapping change.

### Component 2: New migration

Generated by `prisma migrate dev --name gitsha_resolutions_resolved_at_precision`. Expected SQL:

```sql
-- AlterTable
ALTER TABLE `gitsha_resolutions`
  MODIFY COLUMN `resolved_at` DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3);
```

**Why this is safe:**

- `DATETIME(0) → DATETIME(3)` is a metadata-only change in MySQL (`ALGORITHM=INSTANT`).
- Table is empty in production (no Plan 5.1 releases have been live yet).
- No FK or index changes — only the column type's precision modifier.
- Backwards-compatible: any `DATETIME` consumer reads both precision levels identically (the value lies within `DATETIME(3)`'s range, just truncated to seconds).

## Data flow (no change)

This plan is metadata-only. The `lookup_branch_sha` and `prune_expired_resolutions` SQL queries that interpolate `NOW() - INTERVAL 7 DAY` against `resolved_at` continue to work — precision-3 vs precision-0 affects only the value being stored, not the comparison semantics.
